For a **Senior Software Engineer** assessment, your architectural decisions are as important as your code. Here is the recommended packaging strategy and core configurations:

#### 1. Proper Packaging (Clean Layered Architecture)
Organize your code to separate concerns and ensure maintainability:

```text
com.govtech.gsrp
├── config             # Configuration beans (Security, OpenAPI, Database)
├── controller         # REST Endpoints (CitizenController, ServiceRequestController)
├── service            # Business Logic Interfaces
│   └── impl           # Service Implementations (keeps business logic separate)
├── repository         # Spring Data JPA Interfaces
├── entity             # JPA Database Entities
├── dto                # Data Transfer Objects (Request/Response payloads)
│   ├── request        # CitizenCreateRequest, ServiceRequestSubmitDTO
│   └── response       # CitizenResponse, ServiceRequestDetailsDTO
├── exception          # Custom Exceptions and GlobalExceptionHandler
├── security           # Security filters, JWT providers, CustomUserDetails
└── util               # Constants, Enums (RequestStatus, ServiceType), Date Helpers
```

#### 2. Core Configurations

**A. Database (application.yml)**
Use a YAML file for better readability. Configure your PostgreSQL/MySQL connection:
```yaml
spring:
  datasource:
    url: jdbc:postgresql://localhost:5432/gsrp_db
    username: ${DB_USERNAME:postgres}
    password: ${DB_PASSWORD:password}
  jpa:
    hibernate:
      ddl-auto: update  # Use 'validate' in production
    show-sql: true
    properties:
      hibernate:
        format_sql: true
```

**B. Security (SecurityConfig.java)**
Implement Role-Based Access Control (RBAC) as per the PDF:
```java
@Configuration
@EnableWebSecurity
public class SecurityConfig {
    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http.csrf().disable()
            .authorizeHttpRequests()
            .requestMatchers("/api/admin/**").hasRole("ADMIN")
            .requestMatchers("/api/agent/**").hasRole("SERVICE_AGENT")
            .requestMatchers("/api/citizen/**").hasRole("CITIZEN")
            .anyRequest().authenticated()
            .and().httpBasic(); // Or JWT
        return http.build();
    }
}
```

**C. API Documentation (Swagger)**
Enable automatic API documentation:
```yaml
springdoc:
  api-docs:
    path: /api-docs
  swagger-ui:
    path: /swagger-ui.html
```

#### 3. Why this approach?
- **Separation of Entities and DTOs**: Never expose your Database Entities directly to the API. This protects your DB schema and prevents over-posting.
- **Service Interfaces**: Allows for easy unit testing (mocking) and future-proofs the app for different implementations.
- **Global Exception Handling**: Ensures the API always returns a structured JSON error response (e.g., 400 Bad Request) instead of a raw stack trace.

### PROMPT
In above do we record and track application properties as YAMl or application.properties

### ANSWER
For a **Senior Software Engineer** role, using **YAML (`application.yml`)** is strongly preferred over `.properties`.

#### Why YAML is better:
1.  **Hierarchical Structure**: YAML allows you to group related properties (like `spring.datasource`) visually. This is much cleaner than repeating the same prefix on every line in a `.properties` file.
2.  **Readability**: Its indented structure is significantly easier to read and maintain as the project grows.
3.  **Reduced Redundancy**: You don't have to re-type keys.
    *   *Properties:*
        ```properties
        spring.jpa.hibernate.ddl-auto=update
        spring.jpa.show-sql=true
        ```
    *   *YAML:*
        ```yaml
        spring:
          jpa:
            hibernate:
              ddl-auto: update
            show-sql: true
        ```
4.  **Standard for Modern DevOps**: Most modern tools (Docker Compose, Kubernetes, CI/CD pipelines) use YAML, so keeping your application configuration in the same format maintains consistency.

#### Implementation:
When you generate the project, simply rename the default `src/main/resources/application.properties` to `application.yml`.

### PROMPT
I need to move postgresql to MySQL, what are the changes that I need to do to change that. Update the code with the requested changes.

### ANSWER
To migrate from **PostgreSQL** to **MySQL**, you need to make the following architectural and configuration changes:

#### 1. Update `pom.xml`
The PostgreSQL JDBC driver must be replaced with the MySQL Connector/J.

**Old Dependency:**
```xml
<dependency>
    <groupId>org.postgresql</groupId>
    <artifactId>postgresql</artifactId>
    <scope>runtime</scope>
</dependency>
```

**New Dependency:**
```xml
<dependency>
    <groupId>com.mysql</groupId>
    <artifactId>mysql-connector-j</artifactId>
    <scope>runtime</scope>
</dependency>
```

#### 2. Update `application.yaml`
The database connection string and driver class name must be updated.

```yaml
spring:
  datasource:
    url: jdbc:mysql://localhost:3306/gsrp_db?createDatabaseIfNotExist=true&useSSL=false&serverTimezone=UTC
    username: root
    password: your_password
    driver-class-name: com.mysql.cj.jdbc.Driver
  jpa:
    hibernate:
      ddl-auto: update
    show-sql: true
    properties:
      hibernate:
        dialect: org.hibernate.dialect.MySQLDialect # Explicitly set for MySQL
```

#### 3. Entity Considerations
Spring Boot and Hibernate handle the mapping automatically for both databases. Since we are using `@GeneratedValue(strategy = GenerationType.IDENTITY)`, no code changes are required in your entity classes, as MySQL supports auto-increment columns natively.
